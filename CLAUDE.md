## 메타 원칙 — 가드레일 vs 지침

스킬·문서·규칙을 작성할 때 다음 두 층을 분리한다:

- **가드레일 (구조적 제약)** — 누가 누구에게 무엇을 주는가, 출력 형식, 모듈 경계, 검증 가능한 산출물 등. **변하지 않으므로 강하게 박는다.**
- **지침 (행동 제약)** — 에이전트가 받은 것을 어떻게 추론·판정·해석하는가. **모델 성능에 따라 필요 없어지거나 다르게 해석될 여지가 있으므로 유연하게 둔다.**

새 규칙을 추가할 때 이 잣대로 먼저 분류:
- 가드레일이면 강제·구체적으로 명시.
- 지침이면 최소한의 권고에 그치고, 강한 행동 처방은 피한다.

행동 처방을 가드레일 (구조 / 투명성 강제) 로 우회 표현 가능하면 그 쪽이 낫다. 예: "X 를 하지 마라" (지침) → "X 를 했다면 명시한다" (가드레일).

## SKILL 작성법

### `$ARGUMENTS` 는 변수가 아니라 substitution point

`SKILL.md` 본문의 `$ARGUMENTS` 는 호출 시점에 실제 사용자 입력으로 **치환된 후** agent 에게 전달된다. agent 가 본문을 읽는 순간에는 토큰이 이미 사라져 있다.

따라서 본문에 `$ARGUMENTS 가 X 면` 같은 조건문을 쓰면, 치환 후 `<실제 입력값> 가 X 면` 형태가 되어 가짜 변수 참조처럼 어색해진다.

본문에서 사용자 입력을 지칭할 때는 자연어 표현을 쓴다:
- ❌ `$ARGUMENTS 가 slug 형태면 그대로 사용한다`
- ✅ `사용자가 제공한 인자가 slug 형태면 그대로 사용한다`

예외: 본문 끝의 `Task: $ARGUMENTS` 컨벤션은 substitution 자체가 목적이므로 그대로 둔다.

## 작업 디시플린 (재발 방지)

세션에서 드러난 패턴에 대한 재발 대책. 행동 처방은 가능한 한 *투명성 강제* 형태로 우회 표현 (메타 원칙에 따라).

### 1. 메커니즘 단언 전 검증

도구·시스템 동작에 대한 단언은 직접 인용 가능한 출처 (시스템 프롬프트 / Skill 도구 description / docs / 코드) 로 뒷받침한다. 출처를 못 대면 "확인 필요" 라고 명시한 채 답하거나, grep / Read 로 확인 후 답한다.

적색 신호: "...로 보입니다 / ...일 것 같습니다 / 그러므로 ... 가능합니다" 식 추론을 추론으로 잇는 표현이 누적될 때 — 검증을 건너뛰고 있다는 뜻.

### 2. `replace_all` 의 의미 분기 명시

같은 토큰을 일괄 치환하기 전, 각 출현 위치가 동일 개념을 가리키는지 한 번 더 확인한다. 한 곳이라도 의미가 갈라지면 `replace_all` 대신 행마다 Edit. 일괄 처리한다면 "모두 동일 개념이라 판단" 이라고 짧게 명시.

특히 치환 대상이 코드 식별자가 아니라 *문장 속 명사* 일 때 거의 항상 의미가 갈라진다.

### 3. 부드러운 권고에는 근거를 묶는다

"자연스럽다 / 맞아 보인다 / 좋을 것 같다" 류 권고에 1줄 근거를 붙인다. 근거가 약하면 "근거 약함, 결정 필요" 라고 명시하고 사용자 판단에 맡긴다.

근거 없는 부드러운 권고는 *판단 부담을 사용자에게 슬쩍 넘기는* 형태로 끝맺어 사후 반박을 유발한다.

### 4. Step-0 — 변경의 최종 목표 확인

이름 변경 / 삭제 / 플래그 토글 등 *기본 사용 패턴을 깨는* 변경 요청은 진행 전 한 번 묻는다: "이게 다음 단계로 이어지는 변화인가, 단발인가?"

특히 변경 후 파일·스킬이 *부분적으로 깨진 상태* (예: frontmatter 의 핵심 필드가 빠진 채) 가 되면 강한 신호 — 사용자에게 후속 계획이 있다는 뜻일 가능성 높음.
