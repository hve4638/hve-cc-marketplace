# inlay

## 목적

에이전트의 코드 작업 규율과 가드레일. 코드베이스의 맥락을 코드 옆에 박아 에이전트가 매 작업마다 진실을 다시 찾는 비용을 차단한다.

## 적용 시점

코드 작성·수정·리팩터·버그 수정·리뷰·모듈 설계 작업에 적용. 중규모 이상 코드베이스, 장기간 진행되는 작업에서 효과 큼.

## 작업 전 판단

코드를 쓰기 전:

1. 가정 명시 — 불확실하면 묻는다. 묵묵히 하나를 고르지 않는다.
2. 다중 해석 표면화 — 요청이 여러 방향이면 모두 제시한다.
3. 더 단순한 대안 — 있으면 말한다.
4. 정지·질문 — 불분명하면 멈추고 무엇이 불분명한지 이름을 붙여 묻는다.

판정: *"이 해석이 틀렸다면 사용자가 다시 작업을 시키게 되는가?" → ✓ 면 확인 후 시작.*

## 작업 시작 절차

적용 시점: 새 모듈 도입, inlay 경계를 넘는 변경, 작업 범위가 불분명할 때. 자명한 in-file 수정은 자동 주입 컨텍스트로 충분.

요청을 받고 코드 탐색을 시작할 때:

1. 1차 inlay 스캔 — 루트에서 `search` 로 1차 inlay 목록과 `purpose` 를 본다. 이번 작업이 어느 1차 inlay 에 닿는지 (또는 닿지 않는지) 추린다.
2. 선정한 inlay 의 entry 진입 — INLAY.md 와 entry 만 먼저 읽는다. 내부 파일은 entry 가 가리키는 부분부터 필요 시 깊이 들어간다.
3. 하위 inlay 로 재귀 — entry 또는 내부 코드가 자식 inlay 의 책임에 닿으면 그 자식에서 1·2 를 반복.
4. 작업 범위·외부 영향 판단 — 어떤 inlay 들에서 변경이 일어나는지, entry 계약이 깨지는 변경인지, 다른 inlay 에 파급되는지 판단한 뒤 시작.

판정: *"이 작업이 닿는 inlay 들의 entry 계약과 내부 책임을 알고 시작하는가?"* → ✓ 면 진행.

## INLAY.md 규칙

코드베이스 루트에 `INLAY.md` 를 둔다. INLAY.md 가 있는 디렉토리는 **inlay**. inlay 탐색 시 INLAY.md 와 진입점 (entry) 을 우선 읽고, 내부 파일은 필요할 때만 깊이 들어간다 — 매 작업마다 inlay 내부 전체를 읽지 않는다. inlay 는 중첩 가능 (부모 안에 자식 inlay).

INLAY.md 고정 구조:

```markdown
---
name: <inlay 이름>
purpose: <책임 한 줄>
entry: <진입점 경로>            # 단일 파일
entry: [<경로>, <경로>, ...]    # 다중 파일
when:
  - <호출 맥락>
---

## 도메인 용어
- **<도메인 용어>** — <정의>
```

- frontmatter 4 필드 (`name`, `purpose`, `entry`, `when`) 고정.
- `when` — 에이전트가 이 inlay 를 언제 호출하는지 고수준 시나리오. 1~3 bullet 권유. 더 많이 써야 한다면 inlay 책임이 비대하다는 신호로 분할 검토.
- 본문 `## 도메인 용어` — **있을 때만**. 도메인 개념 = inlay 내부에서 의미가 고정된 이름. 일반 단어·언어 관례로 닫힌 식별자 제외. 자식 inlay 의 도메인 용어는 자식 INLAY.md 에만 적는다.

분할 판정: *"내부 변경/실패가 entry 를 거치지 않고 inlay 바깥에 영향을 주는가?"* → ✓ 면 경계가 잘못 그어진 신호.

분할: 분할 권고가 보고되면 사용자에게 분할 작업을 제안한다. 사용자가 승인하거나 명시적으로 요청한 경우에만 inlay 를 분할한다. **변경과 분할을 한 번에 섞지 않는다**.

동기화: inlay 내부 작업 시 진입점·도메인 용어가 변경되면 INLAY.md 도 함께 갱신. 깨지는 변경이 필요하면 INLAY.md 와 변경 코드를 한 PR 로 처리.

## entry 규칙

entry 는 inlay 외부 계약의 단일 진실원. 에이전트가 entry 만 보고 호출·검토할 수 있도록 다음을 박는다:

1. 노출 심볼: 외부에 노출되는 모든 함수·클래스·타입·상수가 entry 의 export 로 통과. entry 에 없는 심볼은 외부에 존재하지 않는다.
2. 시그니처: 정적 타입 언어는 타입으로, 동적 타입 언어는 type hint / JSDoc.
3. 부수 효과: DB·네트워크·파일·로그·env·시간 등 시그니처에 안 잡히는 효과는 docstring 으로.
4. 에러 모델: 발생 가능 예외와 의미를 docstring 으로 (또는 `Result<T, E>` 같은 타입으로).

비자명한 사전·사후조건·불변량·외부 의존성·동시성 모델도 해당 시 docstring 에 명시.

진입점은 inlay 디렉토리 안의 소스코드 파일. 외부 계약이 한 파일에 담기면 단일 경로로 적고 (예: `entry: index.ts`), 여러 파일에 분산되면 배열로 나열한다 (예: `entry: [routes.ts, schema.ts]`). 별도 마크다운 인터페이스 정의서는 두지 않는다.

entry 다중 파일 판정: *"각 entry 파일이 같은 외부 계약의 다른 면인가?"* → ✗ 면 inlay 책임이 비대하다는 신호로 분할 검토.

## 도메인 용어 충돌 규칙

레포 내 모든 INLAY.md (부모/자식/형제 무관) 에서 다음을 위반으로 본다:

- 중복: 같은 도메인 용어가 같은 의미로 둘 이상의 INLAY.md 에 정의됨. 정의 위치는 그 용어를 참조하는 모든 inlay 를 포함하는 가장 좁은 컨텍스트로 한정 (자식 안에 머무르면 자식에, 자식 밖까지 미치면 더 상위에). 다른 위치에서는 제거.
- 충돌: 같은 도메인 용어가 다른 의미로 두 INLAY.md 에 등장. 둘 중 하나 또는 둘 다 이름을 바꾸거나 정의를 통합한다.

한 inlay 내부에서는 하나의 개념에 하나의 이름만 쓴다. 외부 API 와 내부 표현이 다른 경우 (예: snake_case 와 camelCase) 진입점 경계에서 명시 매핑하고, 매핑 이후 내부에서는 단일 이름으로 통일한다.

## WHY 주석 규칙

형식: `// WHY: 이유`. 언어별 주석 기호로 시작한다 (`#`, `/* */` 등). 다중 라인은 첫 줄에만 태그, 다음 줄부터 들여쓴다.

```ts
// WHY: 공유 상태를 갱신하므로 병렬 처리하면 race condition 이 난다.
//      순차 실행을 유지해야 한다.
```

WHY 주석을 쓰기 전, 먼저 다음 한 문장을 완성한다:

> "이 코드만 봐선 독자가 ___ 를 모른다."

빈칸을 **구체적 사실 한 가지** 로 채울 수 있으면 → 그 사실이 곧 WHY 본문이다.
빈칸을 못 채우거나 추상 표현 ("안전을 위해", "X 를 막기 위해" 등) 만 떠오르면 → **주석을 쓰지 않는다.**

빈칸이 자주 취하는 모양:

1. 외부 시스템·라이브러리가 ___ 처럼 동작한다
2. 과거에 ___ 한 버그·이슈가 있었다
3. ___ 한 단순한 대안이 있지만 ___ 때문에 거절했다
4. ___ 한 불변성이 ___ 시점에 유지된다 (또는 깨진다)
5. 보기엔 A 같지만 실제로는 B 다
6. 성능·보안·호환성이 ___ 한 제약을 강제했다

이 모양 어디에도 안 맞아도 빈칸이 구체적이면 WHY. 모양 중 하나처럼 보여도 빈칸이 추상적이면 WHAT.

**빈칸이 안 채워지는 신호**:

- "안전을 위해..." → 무엇이 안전한지 빠짐
- "X 면 skip" → 코드 조건문의 한국어 번역
- "X 를 막기 위함" → 분기의 직역
- "Y 기록", "Y 호출" → 함수 호출의 직역

`WHY:` 가 붙은 라인을 수정·삭제할 때는 **주석의 조건이 아직 유효한지 먼저 검증**한다. 유효하지 않게 되었다면 주석과 코드를 함께 갱신. 주석만 남기고 코드를 바꾸는 일 금지.

## 변경 범위 제한

WHY 주석이 없는 라인이라도 요청 범위 밖이면 건드리지 않는다:

- 인접 코드·주석·포맷을 "개선" 하지 않는다.
- 깨지지 않은 것을 리팩터하지 않는다.
- 기존 스타일을 유지한다 (본인 취향으로 바꾸지 않는다).
- 무관한 죽은 코드는 **언급만**, 삭제하지 않는다.
- 본인 변경이 만든 orphan (미사용 import·변수·함수) 은 제거한다. 기존 dead code 는 요청 없으면 유지한다.

판정: *"변경된 모든 라인은 사용자 요청으로 직접 추적되는가?"*

## 자동 inlay 주입

플러그인이 등록한 훅이 `Read` / `Edit` / `Write` / `MultiEdit` / `NotebookEdit` 호출을 가로채 해당 파일의 조상 INLAY.md 체인을 자동으로 system-reminder 에 주입한다.

- 주입 형태: `<inlay-context path="...">...</inlay-context>` 블록을 루트 → 리프 순서로 나열.
- 같은 inlay 를 같은 세션에서 두 번째로 만나면 본문이 변하지 않은 한 출력에서 빠진다 (silent skip).
- INLAY.md 자체를 수정하는 호출 (`Edit`/`Write`/`MultiEdit`/`NotebookEdit` + 대상 파일이 `INLAY.md`) 에서는 chain 주입을 막아 자기 본문이 prompt 에 다시 박히는 순환을 차단.
- 도구 실행 후 (PostToolUse) 에도 훅이 작동한다 — INLAY.md 자체 수정이면 새 본문으로 해시 갱신, 내부 파일 수정이면 가장 가까운 조상 INLAY.md 의 mtime 을 추적.

이 자동 주입은 명시 도구 호출과 별개로 작동하므로, 통상 흐름에선 도구를 직접 호출하지 않아도 된다. 명시 호출이 필요한 경우는 아래 도구 절을 참고.

## 도구 사용 규칙

INLAY.md 직접 참조 대신 다음 도구를 사용한다:

- **MCP `search`** — 특정 위치에서 현재 및 자식 한 레벨까지 INLAY.md 탐색 (더 깊은 중첩은 무시). 결과: `name`, `purpose`, `path` 의 JSON 배열. 도구 이름: `mcp__inlay__search`. 인자: `{ path?: string }` (생략 시 cwd). 호출 시점: 작업 시작 절차 step 1 의 1 차 inlay 스캔. 더 깊은 트리는 step 3 재귀 시 한 단계씩.
- **MCP `read_context`** — 특정 경로의 조상 INLAY.md 체인을 루트까지 읽음. 출력은 `<inlay-context path="...">...</inlay-context>` 블록의 NL 문자열, top-down 순. 동일 해시로 이미 제공된 INLAY.md 는 본문 자리에 `(already read)` placeholder 가 박힘 (MCP 서버 프로세스 수명). 도구 이름: `mcp__inlay__read_context`. 인자: `{ path?: string }` (생략 시 cwd). 호출 시점: 편집 전에 *다른* inlay 의 컨텍스트를 미리 보고 싶을 때 (작업 시작 절차 step 2 의 prospective 조회). 통상 in-file 흐름은 자동 주입이 처리.
- **정적 스크립트 `doctor`** — frontmatter 4 필드 (`name`/`purpose`/`entry`/`when`) 깨짐, `when` 빈 list, 본문 300줄 초과 검사. 위반 시 처방을 담은 `<inlay-instruction>` 블록 출력. 실행: `node $CLAUDE_PLUGIN_ROOT/scripts/context-doctor.mjs [<루트>]`. 인자 없으면 cwd. exit 0. 호출 시점: INLAY.md 작성·수정 직후 검증.

## 도메인 용어

- **inlay** — `INLAY.md` 가 있는 디렉토리. 탐색 시 진입점·INLAY.md 를 우선 읽는 단위.
- **1차 inlay** — 루트 inlay 의 자식 중 가지를 따라 처음 만나는 inlay. *N차 inlay* 는 (N-1) 차 inlay 의 자식 중 가지를 따라 처음 만나는 inlay.
- **진입점** — inlay 의 외부 경계 파일. INLAY.md frontmatter `entry` 필드에 경로가 기록됨.
- **의도 주석 / `WHY:` 주석** — "왜 이렇게 했는가" 를 기록한 주석.

</inlay>
